# Performance Report — Search Typeahead

All numbers below are measured from the Go implementation in `backend/`.

## Environment & dataset

| | |
|---|---|
| Backend | Go 1.26, single binary, in-process cache (3 logical nodes × 150 vnodes), SQLite (`modernc.org/sqlite`) |
| Dataset | **150,000 distinct queries**, heavy-head Zipf-like counts (generated by `cmd/gen`; the real Wikipedia loader is `scripts/fetch_dataset.sh`) |
| Index build | Trie + trending seed from 150k rows in **< 1 s**; CSV → SQLite load in **0.9 s** |
| Load generator | `cmd/bench` — 50 concurrent workers, prefixes derived from the dataset (short prefixes naturally hot) |

### Reproduce

```bash
cd backend
go run ./cmd/gen  -n 150000 -out data/bench.csv
go run ./cmd/load -data data/bench.csv -db data/bench.db
go build -o bin/server ./cmd/server && ./bin/server -db data/bench.db -addr :8090 &
go run ./cmd/bench -addr http://localhost:8090 -data data/bench.csv -c 50 -d 10s -mode count
go run ./cmd/rebalance
```

## 1. Latency & throughput

Two views: **client-side** end-to-end (includes HTTP + Go client contention under 50 concurrent workers on localhost) and **server-side** (the suggestion API's own work, from `/metrics`).

| Mode | Throughput | Client p50 | Client p95 | Client p99 | Server p50 (cached) | Server p95 (cached) |
|---|---|---|---|---|---|---|
| count (basic) | **30,546 req/s** | 392 µs | 6.29 ms | 17.8 ms | **25 µs** | **25 µs** |
| recency (enhanced) | 30,319 req/s | 404 µs | 7.02 ms | 18.3 ms | 25 µs | 25 µs |

- The suggestion API serves a cached prefix in ~25 µs (server-side p50/p95). The client-side p95 of a few ms is HTTP round-trip plus Go `net/http` client/GC contention at 30k req/s on one box, not the index.
- Recency mode costs little vs count: the per-request re-rank is over ≤50 candidates (microseconds), so throughput is nearly unchanged.
- Server-side cold (cache-miss) latency: p50 25 µs, p95 100 µs, p99 400 µs. Even a miss (Trie walk + candidate union + re-rank) is sub-millisecond.

## 2. Cache hit rate (consistent-hash distributed cache)

From `/metrics` after the count run:

```
hit_rate = 0.9987   (367,847 hits / 477 misses)
per-node entries: cache-0=179, cache-1=141, cache-2=153   ← keys spread across the 3 nodes
```

99.87% hit rate: the working set of hot prefixes is small and stays cached, so misses are first-touch only. Entries are distributed across the three logical nodes by the consistent-hash ring.

## 3. Write reduction (batching)

6,000 `POST /search` across 12 distinct queries, then `/metrics`:

```
received: 6000   upserts: 144   flushes: 12   dropped: 0   reduction_factor: 41.7×
```

6,000 searches collapsed into 144 DB UPSERTs, a 41.7× write reduction, with zero dropped. (Smaller bursts show proportionally: an earlier 90-search test gave 15×.) The ratio grows with traffic concentration, which is the typeahead workload.

**Failure trade-off:** a clean shutdown flushes the buffer (the server logs `flushed on exit`); a crash loses only the unflushed tail (bounded by buffer size + batch size), and the Trie/recency rebuild from SQLite on restart.

## 4. Database read/write counts

```
db.reads  = 150,000   (one-time: rows scanned to build the Trie at startup)
db.writes = 144       (runtime UPSERTs — the batched writes above)
```

The hot path never touches the DB (cache → Trie), so runtime reads are 0; the only reads are the startup rebuild. Writes are batched, hence 144 for 6,000 searches.

## 5. Consistent hashing (`cmd/rebalance`, 100k keys)

```
Distribution over 3 nodes (MD5, 150 vnodes):
  node-0  33.1%   node-1  34.5%   node-2  32.4%      ← ~1.04× skew (even)

Add a 4th node — keys moved: 24.9%   (≈ 1/(N+1))
Baseline `hash % N` (3→4)  — keys moved: 75.1%
```

Adding a node remaps only ~25% of keys vs ~75% for naive `hash % N`, a 3× reduction in cache churn, and the distribution stays even (validating the MD5-over-FNV choice; FNV-1a measured 1.76× skew).

## 6. Ranking: basic vs enhanced (same `/suggest`, prefix `ja`)

After `java` and `java tutorial` were each searched 500× (the write burst):

| Rank | `mode=count` (all-time) | `mode=recency` (recency-aware) |
|---|---|---|
| 1 | jacket salad dessert — 19657 | **java tutorial — 995** |
| 2 | java jacket javascript — 8067 | **java — 995** |
| 3 | java toaster mercedes — 7705 | jacket salad dessert — 9.9 |

Count mode ranks by all-time popularity; recency mode promotes the recently-searched `java`/`java tutorial` to the top while the high-count-but-cold `jacket salad dessert` drops. Same API; `?mode=` switches the behavior. `GET /trending` returns the burst-searched queries by recency.

## 7. Real Wikipedia dataset (cross-check)

The numbers above use the synthetic generator. Re-running on real English Wikipedia pageviews (`scripts/fetch_dataset.sh`, top 117,182 article titles by views for one hour) gives consistent results:

| Metric | Synthetic 150k | **Wikipedia 117k** |
|---|---|---|
| Throughput | 30,546 req/s | **27,120 req/s** |
| Server cached p50 / p95 | 25 / 25 µs | **25 / 50 µs** |
| Cold (miss) p95 / p99 | 100 / 400 µs | 100 / 400 µs |
| Cache hit rate | 99.87% | **94.4%** |
| Client p50 / p95 | 392 µs / 6.3 ms | 628 µs / 6.6 ms |

The lower hit rate on Wikipedia is expected: 117k diverse real titles give a larger, more spread working set of hot prefixes (each cache node held ~6k entries vs ~150 on the head-heavy synthetic set), so more first-touch misses, yet still 94%+, with server-side latency holding ~25–50 µs. Keys stayed evenly distributed across the three nodes (6819 / 5602 / 5782).

> Reproduce: `./scripts/fetch_dataset.sh 20240101-120000 data/wiki_full.csv && sort -t, -k2,2 -nr data/wiki_full.csv | head -150000 > data/wiki.csv && go run ./cmd/load -data data/wiki.csv -db data/wiki.db`
